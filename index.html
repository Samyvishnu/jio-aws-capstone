Program Charter

Version 1.0

Executive Summary

This program builds a Jio-scale, production-ready live OTT streaming platform on AWS — capable of serving 50M+ concurrent sessions at 76 Tbps. The architecture covers Multi-CDN fan-out, AI-assisted pre-scaling via SageMaker, CMAF low-latency encoding, graceful degradation under traffic storms, and full observability. Delivered by 17 engineers across 6 cross-functional teams over 12 weeks (6 × 2-week sprints), with a simulation-first approach to keep AWS costs under $300 during development.

Business Objectives

  • Stream IPL-scale live events with <4s glass-to-glass latency
  • Support 50M concurrent sessions without degradation
  • Zero-downtime during traffic storms via graceful degradation to S3 static fallback
  • AI-driven pre-scaling 30 min before event kick-off
  • Reduce CDN cost by 35% via intelligent multi-CDN routing
  • Achieve 99.99% uptime SLA for the streaming origin
  • Full CI/CD pipeline with <15 min deployment cycle
  • Security-first: WAF, DDoS protection, least-privilege IAM

Scope Definition

In Scope

  • VPC networking (6 subnets, 2 AZs, NAT Gateway)
  • IAM roles for EKS, Lambda@Edge, SageMaker, CodeBuild
  • EKS cluster (Minikube sim → AWS production)
  • CMAF video pipeline (FFmpeg sim → MediaLive + MediaPackage)
  • CloudFront multi-CDN + Lambda@Edge routing
  • Route 53 health checks + DNS failover
  • SageMaker AI traffic prediction model
  • DynamoDB (sessions + content metadata)
  • Kinesis Data Streams (viewer events)
  • ElastiCache Redis (session state)
  • CloudWatch dashboards + alarms + X-Ray tracing
  • WAF + Shield Standard + Secrets Manager
  • JMeter load test + chaos engineering validation
  • CodePipeline + CodeBuild + ECR CI/CD

Out of Scope

  • VOD (Video on Demand) workflows
  • DRM (Digital Rights Management) integration
  • Multi-region AWS deployment (single region: ap-south-1)
  • Mobile application development (iOS/Android)
  • Billing and payment microservices
  • User authentication beyond demo session tokens
  • Real MediaLive channels in dev (FFmpeg sim used)
  • Production traffic (dev/staging only)
  • GDPR / data residency compliance

Constraints

  • AWS Free Tier limits govern dev environment usage
  • $5 billing alarm is a hard stop; EKS deleted after practice
  • All sensitive credentials via Secrets Manager only
  • Simulation-first: no MediaLive/EKS until Phase 3/4 AWS blocks

Design Technical Design Review — Day 1, 14:00–15:00

Design Topics
  • Network topology — CIDR allocation: VPC 10.0.0.0/16, 6 × /24 subnets. Justify 2 AZs (cost vs 3 AZs)
  • Subnet purpose segmentation — Public (ALB, NAT), Private (EKS nodes), Data (DynamoDB endpoints)
  • NAT Gateway vs NAT Instance — Decision: use NAT Instance (t2.micro, free tier) for dev to avoid $0.045/hr NAT Gateway charge
  • Security Group design — ALB-SG (80/443 inbound), EKS-SG (10.0.0.0/16 inbound), DB-SG (5432/6379 from EKS-SG only)
  • IAM role boundary — EKS cluster role, EKS node role, Lambda@Edge role, SageMaker role, CodeBuild role. No wildcard (*) actions
  • Tagging strategy — All resources: Project=jio-capstone, Environment=dev, Owner=TPM, ManagedBy=awscli

Design Multi-CDN Architecture + AI Pre-Scale Logic

Lambda@Edge Routing Logic
// CloudFront Viewer Request trigger
if (uri.startsWith('/stream/')) → route to
primary CDN (CloudFront, high-perf)
if (uri.startsWith('/score/') ||
uri.startsWith('/promo/')) → route to
secondary CDN (S3 direct, cost-optimized)
on 5xx from origin → serve
S3 static fallback asset
AI Pre-Scale Workflow
  • T-30 min before event: SageMaker model invoked with scheduled event data
  • Model predicts concurrent viewer count (historical IPL data + time-of-day features)
  • Prediction written to SSM Parameter: /jio/expected-viewers
  • EKS HPA reads parameter; pre-scales to required replicas
  • TPM manual gate: TPM must approve scale-up before execution

Common Conflict Scenarios & Resolution Scripts

Technical: "My approach is better" Deadlock

Scenario: Platform Lead wants NAT Gateway; Network Engineer insists on NAT Instance.

Resolution: TPM facilitates decision using 3 criteria: cost, availability, maintenance. NAT Gateway = $32/mo but managed; NAT Instance = $0 (free tier) but needs manual HA. For dev environment: NAT Instance wins. Document in ADR-007. Decision is final; no reopening without new evidence.

Script: "We've heard both sides. Given our dev environment constraint of Free Tier, NAT Instance gives us zero cost and full control. We'll note in the ADR that production would use NAT Gateway for HA. This decision is logged and closed."

Timeline: Team underestimates story complexity mid-sprint

Scenario: EKS cluster setup (S2-04, 8 SP) is now day 5 and still not complete; team lead says it needs 3 more days.

Resolution (TPM actions): 1) Pull 1 engineer from low-priority story to pair on EKS. 2) Descope S2-08 (ALB Ingress) to Sprint 3. 3) Update sprint board; communicate to stakeholders same day. 4) Retro action: add spike task for complex AWS setups in future sprints.

Interpersonal: Passive-aggressive behavior in stand-ups

Scenario: Media Lead consistently says "blocked by Platform" in stand-ups without specifics; Platform Lead becomes defensive.

Resolution: TPM has a 1:1 with each lead separately within 24h. Establish: "In stand-ups, blockers must include: what you need, from whom, by when." Create a #blockers Slack channel with a form: [BLOCKER] [NEEDS X] [FROM Y] [BY Z]. Remove the emotional charge by making it process-driven.
